## Why: a ceiling effect, not a broken experiment

Before concluding "RAG made no difference," it's worth ruling out that grounding silently did
nothing. It didn't — retrieval was checked directly against the raw `retrieved.json` written per
case:

- **check17** (guideline 5.1.1, privacy policy contradiction): top retrieved section was **5.1.1
  itself**, similarity 0.71 — the exact correct guideline text.
- **check05** (guideline 2.2, beta language in release notes): top retrieved section was **2.2
  itself**, similarity 0.73 — again the exact correct guideline text.

So the pipeline works end-to-end: real guideline text was fetched, embedded, retrieved by
semantic similarity, and correctly matched to the check it was grounding. The reason the F1
numbers are identical is that **this specific 21-case dataset was already at the F1=1.00 ceiling
before this experiment** — every previously committed baseline in `docs/llm-scorecard.md`
(Sonnet 5, Opus 4.8, Fable 5, across multiple run dates) already scores 1.00 Tier-A F1 and 0.00
Tier-B FP rate. There is no headroom left for grounding to visibly improve on this dataset; a
model that already reasons correctly from its own knowledge on these fixtures cannot score higher
than 1.00 by also being shown the source text.

## What this does and doesn't show

- **Does show:** the retrieval pipeline (ingest → embed → pgvector → top-k retrieval → prompt
  injection) is correctly wired and returns the right guideline text for the right check.
- **Does not show:** whether grounding changes Pierre's behavior on harder cases — this dataset's
  cases are deliberately minimal fixtures isolating one check each, not adversarial or ambiguous
  ones. A dataset with cases where the *correct* answer depends on a guideline detail the model
  might misremember, or on a section whose wording has genuinely changed since the model's
  training cutoff, would be needed to measure a real effect. None of the current 21 cases were
  designed to test guideline-drift sensitivity specifically.
- **No claim is made** that RAG grounding helps or doesn't help Pierre in general — only that it
  made no measurable difference on this dataset, for the reason above.

## Possible follow-up (not started)

Add cases to `eval/dataset/cases/` designed to be sensitive to guideline drift or to
parametric-memory error — e.g. a case whose expected verdict only makes sense against the
*current* wording of a section that has been revised since training, or a case testing a
guideline number that's easy to confuse with a similarly-worded neighbor. Re-run both
configurations against the expanded dataset and compare again. This is a real next step, not
committed to here.
