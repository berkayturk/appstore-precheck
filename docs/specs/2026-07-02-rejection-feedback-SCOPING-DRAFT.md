# Rejection-outcome feedback loop — SCOPING DRAFT

**Status: DRAFT SCOPING — pre-brainstorm, not approved.** This is a menu of options + a
recommendation + open questions for BT to react to, not a spec to build from. No code, tests, or
manifests were touched to produce this document. Roadmap item **#3** ("rejection-outcome feedback
loop").

**Date:** 2026-07-02

## 1. Problem / value

The tool already publishes two independent, honestly-scoped measurements in `docs/scorecard.md`
(generated by `scripts/scorecard.sh`):

- **Synthetic precision** (`corpus/synthetic/labels.json`) — the project's own fixtures, each
  tagging the rule-ids it must fire (`expect_fire`) and must stay silent on (`expect_absent`).
  Measures **intended-behaviour fidelity against fixtures this project wrote.**
- **Real-panel FP rate** (`corpus/real/manifest.json` + `corpus/real/labels.json`) — 18
  permissively-licensed OSS iOS apps, pinned by commit, scanned, with a one-time human TP/FP
  labelling pass on each finding. Measures **the false-positive rate on real, unrelated code.**

Both are explicit, in `docs/scorecard.md`'s own words: **"Neither measurement claims agreement
with Apple's actual review decisions."** That sentence is the gap roadmap #3 targets. Today the
tool cannot say anything — even a small, heavily caveated thing — about whether its FAIL/WARN
findings actually correlate with what Apple's human reviewers reject apps for. A "rejection
outcome feedback loop" would let real submission outcomes (approved / rejected + which guideline
Apple cited) become a third, independently-labelled measurement axis, closing that gap honestly
rather than by assertion.

**Why this depends on accumulated data, not code:** synthetic and real-panel data are both
available in bulk on day one (write fixtures, clone 18 repos). Real Apple outcomes cannot be
manufactured — they arrive one submission at a time, over weeks (App Store review latency), from
whoever actually submits an app that was scanned by this tool. Shipping the plumbing (schema +
ledger + scorecard section) produces **zero measurement value until outcomes exist**; the value
curve is flat at zero until real submissions start landing, then grows slowly and irregularly.
This is unlike every other roadmap item so far (#2a/b/c), which delivered value the moment the
code shipped. Any proposal here has to be honest that it is building an **empty jar with a label
on it**, and address how that's still worth doing (see §5, bootstrap).

## 2. Hard constraints (recap, must hold in any option below)

1. **Read-only identity.** The tool never modifies the *user's* project. Writing to the tool's own
   measurement corpus is the existing precedent (`.precheck-pass` token, `guidelines-fingerprints.json`)
   — an outcomes ledger is the same category of self-owned state, not a violation.
2. **No competitor name anywhere.**
3. **Honesty constraint (core identity).** The tool publishes *measured* precision/recall and is
   explicit it is a static heuristic, not a guarantee. Real-outcome data will be **sparse (small
   n) for a long time** — the entire design has to resist the temptation to compute a confident-
   looking percentage from 2 or 3 data points. Must distinguish real Apple outcomes from
   heuristic/candidate labels at every point they could be confused.
4. **Offline/zero-dep/byte-identical CLI scan path untouched.** `scan.sh`, `bin/cli.js`,
   `action.yml`, `install.sh` must not change behavior. This is purely a maintainer/reporting-side
   addition, like `scorecard-real.sh` — never sourced by `scan.sh`, never network/data-dependent at
   scan time.
5. **Verdict determinism untouched.** `verdict.sh`'s GREEN/YELLOW/RED math (FAIL/WARN counts of the
   *current* scan) must not be influenced by historical outcome data. A user's verdict on their
   own repo, right now, cannot depend on what happened to *other* apps' submissions — that would
   make the tool's core behavior non-deterministic/non-reproducible, its single biggest behavioral
   guarantee today (see `scorecard.sh`, `findings.sh`, `verdict.sh` — all pure functions of the
   current tree).

## 3. What "mapping an outcome to a finding" actually means

Apple's rejection notices cite one or a small number of guideline numbers (e.g. "3.1.2") in the
Resolution Center — not a rule-id from this tool's internal 42-rule catalog (`rule_slug()` in
`skills/appstore-precheck/scripts/findings.sh`). So an outcome record cannot auto-join to a finding
the way `scorecard-real.sh` joins on `<app>|<rule_id>|<file>|<line>|<commit>` — that key format
only exists because the *real-panel* labels are keyed off findings the scanner itself already
emitted deterministically. A rejection outcome needs a **manual mapping step**: a human (submitter
or maintainer) records Apple's cited guideline number(s), and separately records which of this
tool's rule-ids (if any) correspond, using the same guideline-number strings already used
throughout the rule catalog and `references/` docs. That mapping is inherently a judgment call, not
something `scorecard-real.sh`'s deterministic join can do.

A second, harder nuance: **Apple typically cites only the reason that blocked the app**, not every
issue it noticed. If the tool's scan produced 3 WARNs and Apple rejected for a 4th, unrelated
reason the tool never flagged, that is an unambiguous real recall gap. But if the tool's scan
produced 3 WARNs and the app was *approved* anyway, that does **not** prove the 3 WARNs were false
positives — Apple simply never had to rule on them. This asymmetry (see survivorship bias, §7)
means "recall against real rejections" is measurable in a limited way; "precision against real
approvals" is not cleanly measurable at all from outcome data alone.

## 4. Approaches

### Approach A — Minimal outcomes ledger + scorecard extension (recommended)

A new `corpus/outcomes/` directory, structurally parallel to `corpus/real/`:
- `manifest.json` or a single `ledger.json` (array of outcome records, schema in §6).
- `README.md` documenting the schema, the privacy rules, and the contribution/review process.

A new `scripts/scorecard-outcomes.sh` (parallel to `scorecard-real.sh`) reads the ledger and prints
a **raw tally only** — no computed precision/recall number below a sample-size floor (see §7).
`scripts/scorecard.sh` gets a third mode, e.g. `--outcomes`, appending a clearly-separated "Real
App Store Outcomes" section to `docs/scorecard.md`, following the existing pattern: non-blocking
in CI (`continue-on-error: true`, same as `--real`), never gating a PR, never touching the
synthetic precision floor check.

Every record requires a one-time human review before merging to the ledger (mirroring the
`corpus/real/labels.json` "human labelling pass" discipline already documented in
`corpus/real/README.md`) — contribution is via PR, not an automated/telemetry channel (the tool
has no telemetry and this should not become its first instance of one).

**Pros:** smallest surface, reuses every existing pattern (separate script, non-blocking CI,
human-reviewed-before-merge, raw-tally-until-labelled discipline already precedented by
`UNLABELED` handling in `scorecard-real.sh`). **Cons:** growth is manual and slow; no tooling to
make submission easier than hand-editing JSON.

### Approach B — Approach A + a submission helper script

Same ledger and scorecard extension as A, plus a maintainer-facing `scripts/record-outcome.sh`
that captures a `scan.sh --format json` snapshot at submission time (auto-populating
`findings_snapshot`, stripping file paths for privacy — see §6), and later appends the Apple
decision when it arrives (`--decision approved|rejected --guideline 3.1.2`). Reduces manual JSON
errors and makes the "capture findings at submission time, append the outcome weeks later"
two-step workflow explicit in tooling rather than convention.

**Pros:** same measurement value as A, better ergonomics, one script to test
(`tests/test-record-outcome.sh` can exercise the JSON-shape logic without needing a real Apple
decision). **Cons:** another script to write, document, and maintain for a feature whose value is
still bottlenecked by outcome *volume*, not tooling — the ergonomics gain is real but secondary.

### Approach C — Full per-finding disposition pipeline (maximal, not recommended yet)

Beyond an app-level outcome, record a disposition for **every individual finding** the scan
produced at submission time (`confirmed-issue` / `not-an-issue` / `inconclusive`), enabling
per-rule (not just per-app) precision/recall against real outcomes — the same shape as
`corpus/real/labels.json`'s per-finding TP/FP keys, extended with an `apple_outcome` field.

**Pros:** richest possible signal, eventually enables claims like "rule X has correlated with N
real rejections, 0 false alarms." **Cons:** per Apple's single-cited-reason behavior (§3), most of
those per-finding dispositions for an *approved* app are unknowable, not just unlabelled — there is
no way to know if Apple would have objected to a WARN it never had to rule on. Building a schema
that implies that knowledge risks the exact overclaiming the honesty constraint forbids. This is
worth revisiting once real outcome volume is large enough that only the "cited reason" subset
(which *is* knowable) carries real weight — premature now.

## 5. Recommended approach

**Approach A now; Approach B as a natural fast-follow once the schema/process is proven; defer
Approach C indefinitely** (revisit only if/when outcome volume is large, e.g. dozens of records,
and a per-rule breakdown becomes meaningful without leaning on unknowable counterfactuals).

Rationale:
- A is the smallest thing that lets the project say something honest and new: "we now track real
  outcomes, here they are, raw, uninflated" — matching the existing `corpus/real/` precedent almost
  exactly (same directory shape, same human-review-before-merge discipline, same non-blocking CI
  job, same "no unreviewed number is ever published" rule already stated in
  `corpus/real/README.md`'s Honesty section).
- Value here scales with **outcomes accumulated**, not engineering effort. Over-building (B or C)
  before a single real outcome exists is solving a problem that doesn't exist yet; A can be
  extended to B/C later without a rewrite, since the ledger schema is additive.
- Keeping this entirely in the reporting layer (new script, extended `scorecard.sh`, `docs/scorecard.md`
  section) means zero risk to `scan.sh`/`verdict.sh`/the offline CLI — the single hardest
  constraint to accidentally violate here, since it would be tempting to eventually feed outcome
  data back into which rules fire or at what severity. That temptation should be named and
  explicitly deferred (see Open Question 5).

## 6. Data schema sketch

One record per real submission outcome, in `corpus/outcomes/ledger.json`:

```jsonc
{
  "id": "2026-07-02-appref01",                  // stable id, not derived from real app identity
  "app_ref": "maintainer-app-1",                 // anonymizable label; NEVER a real bundle id,
                                                  // company, or app name unless the submitter
                                                  // explicitly opts to disclose it
  "precheck_version": "v1.9.0",                  // PRECHECK_VERSION at scan time
  "scan_date": "2026-06-15",
  "verdict_at_submission": "YELLOW",             // from verdict.sh at the time of the real submission
  "findings_snapshot": [                         // from `scan.sh --format json`, file/line stripped
    { "rule_id": "subscription-links-restore", "severity": "WARN", "guideline": "3.1.2" },
    { "rule_id": "screenshots-per-locale",      "severity": "WARN", "guideline": "2.3.3" }
  ],
  "submission_date": "2026-06-16",               // date submitted to App Store review
  "apple_decision": "rejected",                  // "approved" | "rejected"
  "apple_decision_date": "2026-06-19",
  "rejected_guidelines": ["3.1.2"],               // Apple's cited guideline number(s); [] if approved
  "matched_rule_ids": ["subscription-links-restore"], // human-mapped: which rule_id(s), if any,
                                                       // correspond to the cited guideline(s)
  "outcome_label": "predicted-and-flagged",       // see taxonomy below
  "evidence_kept_privately": true,                // whether the submitter retains (off-repo) evidence
  "reviewed_by": "bt",                            // maintainer who reviewed before merge
  "notes": "free text, caveats, ambiguity in the mapping, etc."
}
```

`outcome_label` taxonomy (app-level, not per-finding — see §3 for why per-finding is deferred):
- `predicted-and-flagged` — Apple rejected for guideline G; the tool had already FAILed/WARNed on a
  rule mapped to G.
- `missed` — Apple rejected for guideline G; the tool had no finding mapped to G (a real recall
  gap; the most valuable/actionable record type).
- `approved-clean` — app approved; tool had 0 FAIL at submission.
- `approved-with-warns-unaddressed` — app approved; tool had 1+ WARN still present at submission
  (suggestive, not proof, that the WARN wasn't blocking — see survivorship-bias caveat, §7).

No field stores Apple's verbatim Resolution Center text — `notes` is free-text summary only;
verbatim correspondence is a privacy/ToS risk (see §8) and is explicitly out of the committed
schema.

## 7. Honesty / reporting plan

- New, clearly separate `docs/scorecard.md` section: "Real App Store Outcomes (n=N)" — never
  blended into synthetic precision or real-panel FP rate; three independently-labelled metrics,
  three independent methodologies, stated as such (extending the existing Methodology section's
  two-corpora framing to three).
- **Below a sample-size floor (proposed: n<10, open question — see §9):** show only the raw tally
  table (`predicted-and-flagged: X, missed: Y, approved-clean: Z, approved-with-warns-unaddressed: W`)
  with a bolded line: *"n=N is too small to compute a meaningful rate; shown for transparency
  only."* No percentage is computed or published at all below the floor — mirrors the existing
  `UNLABELED` discipline ("no unreviewed number is ever published").
- **Above the floor:** may show a directional recall estimate (e.g. "flagged the cited guideline in
  X of Y real rejections") using the same "directional/candidate, not a guarantee" language the
  project already uses elsewhere (per current n=1-honesty convention on other candidate labels),
  plus a permanent, non-removable **survivorship-bias caveat**: apps that saw RED/FAIL locally and
  were fixed before submission never produce a rejection record for that FAIL, so the "approved
  despite WARNs" bucket cannot validate FAIL-level severity choices — only WARN/PASS-level ones.
- CI: always non-blocking (`continue-on-error: true`), like `--real` — outcome data arrives
  asynchronously and manually, and must never gate a build or a synthetic-precision floor check.
- Never wired into `verdict.sh` or `scan.sh` — this is a reporting/methodology layer only, exactly
  like `scorecard-real.sh` today. If real-outcome data eventually informs *which rules exist or what
  severity they carry*, that is a maintainer/roadmap decision made by a human reading the scorecard,
  the same way `guideline-drift.sh` findings are a human/agent triage step, not an auto-applied
  change (see `docs/specs/2026-07-02-semantic-guideline-drift-design.md`).

## 8. Bootstrap: what "day one" looks like

The ledger starts empty (`[]`), exactly like `corpus/real/labels.json` started `{}`. First value
comes from the single most available honest source: **the maintainer's own app(s)**, if any go
through App Store review while this feature exists — recorded with full disclosure since it's the
maintainer's own data to share. That first entry should be published with maximal caveats
("n=1, one data point, directional only") — consistent with the project's existing practice of
labelling early/thin measurements as candidate/directional rather than waiting for statistical
significance to say anything at all. A low-friction path for **other users** to contribute (e.g. a
GitHub issue template prompting for the schema fields, funneled into a maintainer-reviewed PR) is
worth having from day one even though initial volume will be near zero — it costs little to build
and removes a barrier for the first external contributor whenever they show up.

## 9. Open questions for BT

1. **Storage: in-repo (public) vs. maintainer-held-privately-with-published-aggregates?** In-repo
   `corpus/outcomes/ledger.json` maximizes auditability (anyone can verify the raw tally behind a
   published rate) but means even anonymized outcome data lives in the public repo forever.
   Keeping raw records private and publishing only aggregate counts sacrifices that auditability —
   users have to trust the maintainer's summary math instead of checking it themselves. Real-panel
   labels chose "in-repo, public" already; is that the right default here too given real Apple
   decisions are more sensitive than OSS-code TP/FP calls?
2. **Contribution model: maintainer-only, or user-contributable?** Self-reported "Apple rejected me
   for 3.1.2" claims are unverifiable and could be gamed (inflating or deflating the tool's
   apparent recall for marketing reasons, in either direction). Is a PR + maintainer-review gate
   (mirroring the existing real-panel labelling pass) sufficient trust, or does this need a
   stronger evidence requirement before a submitted record is merged?
3. **Sample-size floor:** what n triggers switching from "raw tally only" to "a directional rate
   with caveats"? The existing synthetic-precision floor (0.80) is a *value* floor; this needs a
   *sample-size* floor instead — is 10 reasonable, too high, too low, given how slowly this data
   will realistically accumulate (see Risks)?
4. **Scope boundary — reporting only, or eventually roadmap-informing?** As scoped here (§7), this
   is purely a `docs/scorecard.md` reporting feature. A legitimate but larger future use: a WARN
   rule that never once correlates with a real rejection across N outcomes becomes a candidate for
   severity downgrade or removal. That's a meaningful scope expansion (measurement → roadmap
   input) — is it in scope for #3, or a distinct future roadmap item once there's enough data to
   act on?
5. **Evidence handling:** should a submitted record require the submitter to retain (privately,
   off-repo) evidence such as an Apple Resolution Center screenshot, in case a record is later
   disputed or needs re-verification? If so, who holds it and for how long, and does that create a
   support/liability burden disproportionate to the feature's value at low n?

## 10. Risks / unknowns

- **Survivorship bias is structural, not fixable by better schema design** (§3, §7): the tool
  cannot observe rejection outcomes for apps whose FAILs were fixed before submission, meaning FAIL
  severity can never be validated against real rejections — only WARN/PASS-level judgment calls
  can be, and only in the "approved anyway" direction, never conclusively.
- **Volume will be small for a long time.** This is a niche tool; App Store review takes days to
  weeks; a realistic growth rate might be single digits per quarter even with active promotion.
  There's a real risk this becomes permanent, mostly-empty scaffolding — worth setting an informal
  revisit/stop-loss point (e.g., "if still n<5 after 6 months, reconsider whether this is worth
  maintaining as a documented feature vs. quietly parking it").
- **Guideline-number → rule-id mapping is a manual, sometimes-ambiguous judgment call** (§3) —
  Apple's ~200 numbered sub-guidelines don't map 1:1 onto this tool's 42-rule catalog; a
  mis-mapping (matching the wrong rule, or claiming "no match" for a rule that actually was
  relevant) is itself a source of measurement error the schema can't fully prevent, only document
  via the free-text `notes` field.
- **No cryptographic or otherwise strong way to verify a real Apple decision** — the design leans
  entirely on maintainer trust + PR review, the same trust model as `corpus/real/labels.json`'s
  human TP/FP pass, but the stakes of a fabricated *rejection* claim (used to inflate or deflate
  the tool's apparent value) are arguably higher than a mislabelled OSS finding.
- **Privacy/confidentiality of Apple Resolution Center content is a real unknown, not just a
  style choice.** Whether developers are free to share rejection reason text is a genuine open
  question under the Apple Developer Program License Agreement in some readings; this scoping
  deliberately keeps verbatim Apple text out of the committed schema (§6) as a precaution, but that
  is a mitigation, not a confirmed-safe legal position — flagging for BT awareness, not resolving
  it here.
- **Public in-repo ledger + small n makes re-identification easier**, not harder, than the 18-app
  real panel (which is OSS code with no confidentiality concerns at all): a rejection tied to a
  specific date/version/guideline combination could be correlated with public information (e.g. a
  developer's public tweet about a rejection) even with `app_ref` anonymized. Anonymization here
  needs to be treated as a real design problem, not just "omit the app name."
