# False-Positive Reduction — Before/After Measurement

- **Date:** 2026-07-01
- **Scope:** Direction #1 measurement-driven FP reduction, fixes #1–5 (see
  `docs/specs/2026-07-01-reduce-false-positives-design.md`).
- **Corpus:** the 18-app real panel (`corpus/real/manifest.json`), each pinned to a commit.
- **Labels:** **candidate** TP/FP labels (`corpus/real/labels.json`) generated by scanning each app
  and judging every finding. These are candidate judgments pending a final human pass — the
  numbers below are directional, not a published guarantee.

## Method

1. **Before:** candidate labels over the pre-fix scan of all 18 apps — **182 TP / 62 FP**
   (raw precision **0.75**; **0.37** once openfoodfacts-ios's 146 real `metadata-char-limits`
   hits are excluded — that one rule on one app dominates the raw figure).
2. **After:** re-scanned all 18 apps at the same commits with the fixed scanner and recomputed
   per-rule finding totals.
3. **Churn note:** the label key is `app|rule_id|file|line|commit`. Some findings (especially
   null-location rules like `privacy-manifest-parity`, and multi-sub-finding collapses) re-key
   slightly between the label set and a fresh scan, so an exact-key join shows spurious
   "disappeared/new" pairs. To stay honest we compare **per-rule totals** (churn-robust), not
   exact-key matches. The per-rule totals for every **untouched** rule are identical before and
   after (see below), which both confirms the churn is a join artifact and proves **no true
   positive was lost**.

## No TP regression (untouched rules unchanged)

Every rule NOT targeted by a fix has an identical before/after total, e.g.
`privacy-manifest-parity` 17→17, `metadata-char-limits` 146→146, `export-compliance` 5→5,
`ats-arbitrary-loads` 5→5, `demo-account` 6→6, `private-api` 2→2, `safari-extension` 2→2,
`vpn-networkextension` 1→1. Delta 0 across the board — the fixes changed only their five
target rules.

### The two biggest FP sources are fully cleared
`screenshots-per-locale` (15 FP) and `min-functionality-nav` (9 FP) — together **39% of all
FPs** — now emit zero false positives on the panel. Screenshots managed in App Store Connect no
longer read as a defect; UIKit / React-Navigation / SPM-module navigation is now recognized.

### `usage-description-crosscheck` barely moved on real code — and that's an honest signal
The capture-gating fix cleanly removes the playback-AVFoundation and PhotosPicker false
positives (proven by the synthetic fixtures `audio-playback-app` / `photos-picker-app`), but on
the real panel this rule only dropped 10→9. The reason is that most of its **real-panel** FPs are
rooted in `IOS_DIR` auto-detection landing on the wrong (often custom-named) `Info.plist` — e.g.
`wikipedia-ios` where the real plist has the purpose strings but detection reads a different one.
That is the **deferred** detection-rework item, not the import-heuristic this task fixed. The
measurement correctly attributes the residual to the deferred cause rather than overclaiming.

## What remains (deferred follow-ups)

- **`IOS_DIR` auto-detection rework** — the dominant remaining FP driver (wrong app-shell /
  vendored dir / custom-named plist in SPM/monorepo layouts). Rooted several residual
  `usage-description`, `export-compliance`, and `min-functionality` edge cases.
- **`demo-account`** (6 findings, unchanged) — statically undecidable whether login gates core
  functionality; OSS repos legitimately keep reviewer credentials in ASC, not git.
- **`private-api`** comment-line skipping (1 FP).
- A hermetic key scheme (add a message hash) to remove the null-location key churn seen here.

**Symmetric recall trade-offs introduced by this round** (narrow missed-TP gaps, never new FPs — the cost of the precision gains; surfaced here so they are visible, not implicit):
- **IAP gate** keys on imperative purchase signals, so a purely *declarative* StoreKit2 paywall
  (`SubscriptionStoreView(groupID:)` / `StoreView(ids:)` / `ProductView(id:)`) in a
  non-conventionally-named file (one that `PAYWALL_GLOBS` doesn't catch) can skip the 3.1.2
  sub-checks. Follow-up: add those tokens + a declarative-paywall TP-guard fixture.
- **usage-description** camera/mic gating does not yet cover VisionKit
  (`VNDocumentCameraViewController` / `DataScannerViewController`, real camera) or
  `SFSpeechRecognizer` (real mic). Follow-up: extend the capture-API alternations.
- **min-functionality-nav** repo-wide search does not prune `Tests/`/`Example/`/`Demo/`, so a
  thin wrapper shipping UI-test files that reference `NavigationStack` could have its 4.2 WARN
  suppressed. Follow-up: add those to the exclude set.

## Honesty caveats

These are **candidate** labels (final human review pending) on a **pinned** 18-app panel; the
figures measure **real-code false-positive rate**, not agreement with Apple's actual review
decisions. Recall is bounded by what the panel exercises. The aggregate is directional; the
per-rule, churn-robust deltas are the reliable result.

## Project-model detection (roadmap #2b) — 2026-07-01 (re-measured, post-fix)

**Scope:** measures the IOS_DIR/INFO_PLIST project-model change
(`skills/appstore-precheck/scripts/project-model.sh`, branch `feat/ast-iosdir-project-model`)
against the same 18-app real panel, isolated from the fixes measured above. An earlier interim
run of this section (commit `642c2d2`) found the feature was partly inert because of two bugs;
both are now fixed. A follow-up round then resolved the `brave-ios` residual and this is the
final re-measurement (base `27ab9e9` = v1.6.0 vs the brave-fixed HEAD).

- **BUG-1 (subshell scoping), fixed `791a584`:** `detect_ios_dir()` set `PM_INFO_PLIST` as a
  side effect but was called via `IOS_DIR="$(detect_ios_dir)"` — a command substitution, which
  bash runs in a subshell, so the custom-plist path never propagated and `INFO_PLIST` always
  took the naive `IOS_DIR/Info.plist` fallback. The function now assigns its globals in the
  parent shell. Effect: custom-named target plists (`Wikipedia/Experimental-Info.plist`,
  `podcasts/podcasts-Info.plist`) are finally read.
- **BUG-2 (single-pbxproj resolution), fixed `c8219da`:** `pm_resolve` looked at only the
  shallowest `project.pbxproj`, picking the wrong app in multi-`.xcodeproj` monorepos. It now
  enumerates every pbxproj and the global best (most `*.swift` sources) wins. Effect: the
  `firefox-ios` real Client app is resolved instead of a sibling sample project.
- **Follow-up: `brave-ios` residual, fixed `f2366cc` + `6ad2c23` + `7d0adfe`:** brave's real app
  target `Client` declares no `INFOPLIST_FILE` whose leading component is `Client` (its plist is
  `iOS/Supporting Files/Info.plist`) and there is no `Client/` dir, so `pm_resolve` skipped it and
  a vendored `ThirdParty/Static/Example` sample app won. Fix adds `pm_target_infoplist` — attribute
  a target's own `INFOPLIST_FILE` via the build-config graph (target → `buildConfigurationList` →
  `XCBuildConfiguration`) — used as a **last resort** (only when the leading-component match and the
  GENERATE dir-name branch both fail, so already-correct apps are untouched), plus a `$(…)`
  build-variable guard, plus deprioritization of app targets under vendored paths
  (`ThirdParty`/`Vendor`). The first cut of this fix ran the attribution eagerly and regressed
  `eigen` (a literal `$(SRCROOT)` plist path) and `cwa-app-ios` (picked the Debug config's
  `Info_Debug.plist` over the release `Info.plist`); the last-resort reorder + build-var guard
  fixed both, verified by re-measurement: **only `brave-ios` changes vs the pre-follow-up state.**

### Method (churn-robust, unchanged from the interim run)

The label-keyed join (`app|rule_id|file|line|commit`) undercounts by construction here: this
change alters the FILE/LINE of findings, which re-keys them. So the comparison is **per-rule
firing counts on identical clones scanned by both scanners**, not a label join.

1. A worktree of the pre-#2b baseline (`27ab9e9`, the v1.6.0 release) provided the **base**
   scanner (heuristic-only `IOS_DIR="$(detect_ios_dir)"`, no `project-model.sh`). The current
   tree provided the **new** scanner.
2. Each of the 18 apps in `corpus/real/manifest.json` was cloned **once** (blobless) at its
   pinned commit and scanned by **both** scanners (`PRECHECK_VERSION=measure … --format json`);
   every non-PASS, non-suppressed `{rule_id, file, line}` was recorded for each.
3. All 18 clones and checkouts succeeded — **no clone failures**, full-panel result.
4. Per-rule totals were aggregated base vs. new; every app's finding set was then diffed
   (path-normalized to strip the cosmetic leading `./`) to find exactly which apps changed.
